- **Complexity Estimate**: Moderate（仕様準拠のテンプレートエンジンをゼロから実装するが、対象範囲は標準機能一式に限定し明確にスコープダウン済み）

## Functional Requirements

### FR-1: Mustache処理エンジンの自作
既存クレート（`mustache`, `ramhorns`等）をラップせず、パーサー・レンダラーを独自に実装する。プロジェクトの目的そのものが処理系の実装であるため。

### FR-2: 提供形態
コア機能（パース・レンダリング）をライブラリとして実装し、CLIはそのライブラリを利用する薄いラッパーとして提供する。

### FR-3: 入力データ形式
テンプレートに埋め込むデータの入力形式として、JSONおよびYAMLをサポートする。TOML等その他形式は対象外とする。

### FR-4: Mustache仕様準拠範囲
公式Mustache仕様の必須（mandatory）機能一式をサポートする:
- 変数展開（エスケープあり `{{var}}` / エスケープなし `{{{var}}}` または `{{&var}}`）
- セクション `{{#section}}...{{/section}}`（真偽判定・配列繰り返しの両方）
- 逆セクション `{{^section}}...{{/section}}`
- パーシャル `{{> partial}}`
- コメント `{{! comment }}`
- デリミタ変更 `{{=<% %>=}}`

**対象外**: ラムダ（`~lambdas`）およびその他のオプション拡張モジュール（`~inheritance`, `~dynamic-names`等）。JSON/YAMLのデータ入力では関数を表現できずCLI用途で活用できないため、v1のスコープからは除外する。将来ライブラリAPIとしての需要が明確になった場合に再検討する。

### FR-5: CLIの入出力インターフェース
- テンプレートファイル・データファイル・出力先ファイルをコマンドライン引数で指定可能とする。
- 標準入力・標準出力によるパイプ処理に対応する。
  - デフォルト: データを標準入力から読み込む（`mustache template.tmpl < data.json`）。テンプレートは常にファイル引数として指定する。
  - オプション指定時: テンプレート側を標準入力から読み込む切り替えが可能。
  - テンプレート・データ双方を同時に標準入力に割り当てることはできない（単一ストリームのため）。その組み合わせが指定された場合はエラーとする。
- 設定ファイルによる一括バッチ変換は対象外（v1スコープ外）。

### FR-6: パーシャルの解決方法
- パーシャルは指定ディレクトリ内のファイルから読み込む。
- パーシャル用ディレクトリは`--partials-dir`オプション等で指定する。相対パス・絶対パスいずれも指定可能で、相対パスは実行時のカレントディレクトリを基準に解決する。
- `--partials-dir`が未指定の場合のデフォルトは、メインテンプレートファイルが置かれているディレクトリとする（実行時のカレントディレクトリではなく、テンプレート一式の相対的な位置関係を保つため）。

### FR-7: 未定義変数の挙動
- デフォルトでは、データに存在しない変数は空文字として扱い、処理を継続する（Mustache仕様のデフォルト挙動）。
- オプション（strictモード等）で、未定義変数の参照をエラーとして処理を中断する挙動に切り替え可能とする。

### FR-8: 著作権・ライセンス表記
生成するすべてのソースコードファイルの先頭に、著作権・ライセンス表記のコメントを含める。
- 著作権者名: `agwlvssainokuni`
- ライセンス: Apache License 2.0（プロジェクトの`LICENSE`ファイルに準拠）
- `LICENSE`ファイル自体（`[yyyy] [name of copyright owner]`のプレースホルダ）は現状のまま変更しない。

## Non-Functional Requirements

### NFR-1: 配布形態
シングルバイナリとしてクロスプラットフォーム配布できることを想定する（`cargo install`、GitHub Releases等での配布）。

### NFR-2: テスト方針
- 標準的な単体テスト・結合テストに加え、公式mustache/specのコンフォーマンステストスイート（JSON形式のテストケース集）を取り込み、仕様準拠を検証する。
- 取り込み対象はFR-4で定めた必須モジュール（comments, delimiters, interpolation, inverted, partials, sections）に対応するテストケースとし、ラムダ等オプションモジュールのテストケースは対象外とする。

### NFR-3: プロパティベーステスト（PBT拡張機能: 有効）
- PBT拡張ルール（PBT-01〜PBT-10）を全面適用する（Partial enforcementではなくフル適用）。
- 特にパーサー・レンダラーはラウンドトリップ性・不変条件の検証に適しており、公式specの例示ベーステストと組み合わせて網羅性を高める。
- 使用フレームワークはNFR Requirementsステージで確定する（Rust向け推奨: `proptest`）。

### NFR-4: セキュリティ拡張機能
対象外（無効）。ローカルCLIツールであり、テンプレート・データとも利用者自身が用意する想定のため、本プロジェクトの現段階ではセキュリティベースラインを強制しない。

### NFR-5: 耐障害性拡張機能
対象外（無効）。単一プロセスで完結するCLIツールのため、HA/DR等の分散システム向け耐障害性設計は本プロジェクトには適用しない。

### NFR-6: コード品質（著作権表記）
FR-8に対応する非機能要件として、全ソースファイルへの著作権・ライセンスヘッダーコメント付与をコード生成時の標準手順に組み込む。

## Summary

RustでMustache処理エンジンを自作し、ライブラリ＋CLI（`mustache`コマンド）として提供する。入力データはJSON/YAML、対応仕様はMustache必須機能一式（ラムダは対象外）。CLIはファイル・標準入出力の両対応とし、パーシャルはディレクトリベースで解決する。未定義変数はデフォルトで継続、strictモードでエラー化可能。配布はシングルバイナリでのクロスプラットフォーム対応を想定し、テストは公式specコンフォーマンススイートとプロパティベーステストで品質を担保する。全ソースファイルにApache License 2.0の著作権ヘッダー（著作権者: agwlvssainokuni）を付与する。
