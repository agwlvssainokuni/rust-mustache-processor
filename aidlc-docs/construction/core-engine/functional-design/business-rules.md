# Business Rules — core-engine

公式mustache/spec（NFR-2）に準拠する規則と、core-engine-functional-design-plan.mdで決定した5つの実装判断（Q1〜Q5）を統合した業務ルールカタログ。各ルールは`domain-entities.md`のエンティティを参照する。

> **Step8（Code Generation, spec conformanceテスト）での重要な補正について**:
> 本カタログの初版は公式spec準拠を意図して作成したが、実際に公式mustache/specのJSONフィクスチャ（`tests/spec/fixtures/`）に対して実装を検証したところ、初版が見落としていた仕様（暗黙のイテレータ`.`、ドット区切り名前）や、誤って導出した規則（BR-2.4のコンテキスト非プッシュ方針、BR-5.2の常時エラー方針、BR-5.5の循環検出方針）が判明した。該当箇所は下記に太字で修正履歴を付記している。詳細な経緯は`aidlc-docs/construction/plans/core-engine-code-generation-plan.md`のStep8補正記録を参照。

## 1. 変数展開ルール（FR-4）

| ルール | 内容 |
|---|---|
| BR-1.1 | `{{name}}`は値をHTMLエスケープして出力する。エスケープ対象: `&` → `&amp;`, `<` → `&lt;`, `>` → `&gt;`, `"` → `&quot;`, `'` → `&#39;` |
| BR-1.2 | `{{{name}}}`または`{{&name}}`はエスケープなしで出力する |
| BR-1.3 | `Value::Null`は空文字として出力する |
| BR-1.4 | `Value::Bool`は`"true"`/`"false"`として出力する |
| BR-1.5 | `Value::Integer`は`i64`の標準的な10進表記（例: `-3`, `42`）として出力する（Q1=B） |
| BR-1.6 | `Value::Float`は`f64`の標準的な10進表記（例: `1.5`, `-0.25`）として出力する。整数値の`Float`（例: `2.0`）も小数点を保持する（Q1=B） |
| BR-1.7 | `Value::String`はそのまま出力する（BR-1.1/BR-1.2のエスケープ規則を適用） |
| BR-1.8 | `Value::Array`/`Value::Object`は変数展開の対象として未定義（spec上も規定なし）。実装は空文字として扱う |
| BR-1.9 | コンテキストスタック上に該当キーが存在しない場合: strictモードOFF（既定）→ 空文字として継続（FR-7）。strictモードON → `RenderErrorKind::UndefinedVariable`（Q2=A、変数展開のみが対象） |
| **BR-1.10（Step8追加）** | `{{.}}`（暗黙のイテレータ）は、現在のコンテキストスタックの最上位（現在のコンテキスト自体）を値として展開する |
| **BR-1.11（Step8追加）** | `{{a.b.c}}`のようなドット区切りの名前は、最初のセグメント（例:`a`）のみコンテキストスタック探索（BR-4.1/BR-4.2）で解決し、以降のセグメント（例:`b`, `c`）は直前で解決した値への直接のキー参照として辿る。データ中に`"a.b"`という一つのフラットキーが存在しても、それを単一キーとして参照することは絶対にない（"Dotted Names are never single keys"） |

## 2. セクションルール（FR-4）

| ルール | 内容 |
|---|---|
| BR-2.1 | `Value::Bool(false)`, `Value::Null`, `Value::Array(空)` → セクション本体を表示しない |
| BR-2.2 | `Value::Array(非空)` → 各要素をコンテキストスタックにプッシュしてセクション本体を要素数分繰り返す |
| BR-2.3 | `Value::Object` → オブジェクトをコンテキストスタックにプッシュしてセクション本体を1回表示する |
| **BR-2.4（Step8で修正）** | 上記以外の真値（`Bool(true)`, 非0の`Integer`/`Float`, 非空`String`）→ **その値自体をコンテキストスタックにプッシュして**セクション本体を1回表示する（Map・配列要素と同様の扱い。`{{.}}`でスカラーセクションの値自体を参照できるようにするため。当初は「元のコンテキストのまま」としていたが、公式spec conformanceテストの"Variable test"で誤りと判明） |
| BR-2.5 | コンテキストスタック上にキーが存在しない場合 → BR-2.1と同様に非表示（「未定義」は単に偽として扱う。Q2=Aによりエラーにはならない） |
| **BR-2.6（Step8追加）** | `{{#.}}`（暗黙のイテレータをセクション名に使う）は、現在のコンテキスト自体を対象にBR-2.1〜BR-2.4の判定を適用する（配列であれば繰り返し、それ以外の真値であれば1回表示） |

## 3. 逆セクションルール（FR-4）

| ルール | 内容 |
|---|---|
| BR-3.1 | セクションルール（BR-2.1〜BR-2.5）の真偽判定を反転する: BR-2.1に該当する場合のみ本体を表示し、それ以外は非表示 |

## 4. コンテキストスタック探索ルール

| ルール | 内容 |
|---|---|
| BR-4.1 | 変数・セクション名の解決は、コンテキストスタックの最も内側（最後にプッシュされた要素）から外側へ順に探索する |
| BR-4.2 | 内側のコンテキストにキーが存在すればそこで探索を終了する（外側に同名キーがあっても内側が優先される） |

## 5. パーシャルルール（FR-6）

| ルール | 内容 |
|---|---|
| BR-5.1 | `{{> name}}`に到達した時点で`PartialResolver::resolve(name)`を呼び出す（遅延評価、Application Design Q3=A） |
| **BR-5.2（Step8で修正）** | 解決に失敗した場合、**strictモードOFF（既定）→ 空文字列として継続する（公式spec準拠、"Failed Lookup"）。strictモードON → `RenderErrorKind::PartialNotFound`を返す**（開発時の検出手段として、FR-7のstrictモードの対象をパーシャルにも拡張）。当初はApplication Design Q3=A/Functional Design Q3=Aに基づき常時エラーとしていたが、公式spec conformanceテストで空文字列が正しいと判明したため修正 |
| BR-5.3 | 解決に成功した場合、得られた文字列を現在のデリミタ設定でParserにより再パースし、現在のコンテキストスタックでレンダリングする |
| **BR-5.4（Step8で修正）** | パーシャルタグがスタンドアロン（行全体を占める）の場合、タグ直前の行頭空白を`indent`として記録する。`indent`は**パーシャルの値展開前のテンプレート文字列自体**の各行（最終行の末尾改行がない場合を除く）の先頭に適用したうえでパース・レンダリングする。当初はレンダリング後の出力に事後適用していたが、値展開によって挿入された文字列内の改行にまでインデントが波及してしまう不具合が公式spec conformanceテスト（"Standalone Indentation"）で判明したため、値展開前の静的テキストへの適用に修正 |
| **BR-5.5（Step8で削除）** | ~~パーシャル名チェーンによる循環検出~~。公式spec conformanceテスト（"Recursion"）により、同名パーシャルの自己再帰（データに基づき自然終端するツリー/リスト構造など）は正当な実装パターンであり、名前の再出現だけで一律にエラーとしてはならないことが判明した。真に無限の再帰に対する安全装置は、NFR Design Q2で決定した`MAX_NESTING_DEPTH`（セクション・パーシャル共通の深度カウンタ）のみに一本化する |

## 6. コメント・デリミタ変更ルール（FR-4）

| ルール | 内容 |
|---|---|
| BR-6.1 | `{{! comment }}`は出力に一切寄与しない |
| BR-6.2 | `{{=<% %>=}}`形式のタグ以降、次のデリミタ変更まで新しいデリミタで以降のタグを解析する |
| BR-6.3 | デリミタ変更の有効範囲は現在パース中のテンプレート内に閉じる。パーシャル解決先の再パース（BR-5.3）は、パーシャル呼び出し時点のデリミタではなく、そのパーシャル自身の内容のデフォルトデリミタ（`{{` `}}`）から再開する |

## 7. スタンドアロンタグの空白トリミングルール

| ルール | 内容 |
|---|---|
| BR-7.1 | セクション開始・終了・逆セクション・パーシャル・コメント・デリミタ変更タグが、前後を空白のみで囲まれ、行全体を占める場合（スタンドアロン）、そのタグを含む行全体（前後の空白と改行）を出力から除去する |
| BR-7.2 | 変数展開タグ（`{{name}}`, `{{{name}}}`, `{{&name}}`）はスタンドアロン判定の対象外（常にインライン扱い）。**同じ行に変数展開タグが1つでも存在する場合、その行全体がスタンドアロン判定から除外される**（他のブロックタグが同じ行にあってもインライン扱いのまま。Step8で判定ロジックを行単位に是正した際に明文化） |
| **BR-7.3（Step8追加）** | 1つの行に複数のブロックタグ（セクション開始・終了・逆セクション・パーシャル・コメント・デリミタ変更）が並び、それ以外のテキストが空白のみである場合、その行全体をスタンドアロンとして扱う（例:`{{#a}}{{/a}}\n`）。当初は1タグごとの局所判定（直前直後のテキストのみ参照）で実装しており、この複数ブロックタグの行を正しく検出できていなかったため、行全体を単位とする判定に是正した |
| **BR-7.4（Step8追加）** | 改行は`\n`だけでなく`\r\n`も1つの改行として認識する（"\\r\\n" should be considered a newline for standalone tags） |

## 8. エラー処理ルール（Q5）

| ルール | 内容 |
|---|---|
| BR-8.1 | `ParseError`/`RenderError`は発生箇所の行番号・列番号（1始まり）を`line`/`column`フィールドに含める |
| BR-8.2 | パーシャル内で発生したエラーは、パーシャルファイル内での位置を報告する（呼び出し元テンプレートの行番号ではない） |
