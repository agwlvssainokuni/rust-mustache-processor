# Business Rules — core-engine

公式mustache/spec（NFR-2）に準拠する規則と、core-engine-functional-design-plan.mdで決定した5つの実装判断（Q1〜Q5）を統合した業務ルールカタログ。各ルールは`domain-entities.md`のエンティティを参照する。

## 1. 変数展開ルール（FR-4）

| ルール | 内容 |
|---|---|
| BR-1.1 | `{{name}}`は値をHTMLエスケープして出力する。エスケープ対象: `&` → `&amp;`, `<` → `&lt;`, `>` → `&gt;`, `"` → `&quot;`, `'` → `&#39;` |
| BR-1.2 | `{{{name}}}`または`{{&name}}`はエスケープなしで出力する |
| BR-1.3 | `Value::Null`は空文字として出力する |
| BR-1.4 | `Value::Bool`は`"true"`/`"false"`として出力する |
| BR-1.5 | `Value::Integer`は`i64`の標準的な10進表記（例: `-3`, `42`）として出力する（Q1=B） |
| BR-1.6 | `Value::Float`は`f64`の標準的な10進表記（例: `1.5`, `-0.25`）として出力する。整数値の`Float`（例: `2.0`）も小数点を保持する（Q1=B） |
| BR-1.7 | `Value::String`はそのまま出力する（BR-1.1/BR-1.2のエスケープ規則を適用） |
| BR-1.8 | `Value::Array`/`Value::Object`は変数展開の対象として未定義（spec上も規定なし）。実装は空文字として扱う |
| BR-1.9 | コンテキストスタック上に該当キーが存在しない場合: strictモードOFF（既定）→ 空文字として継続（FR-7）。strictモードON → `RenderErrorKind::UndefinedVariable`（Q2=A、変数展開のみが対象） |

## 2. セクションルール（FR-4）

| ルール | 内容 |
|---|---|
| BR-2.1 | `Value::Bool(false)`, `Value::Null`, `Value::Array(空)` → セクション本体を表示しない |
| BR-2.2 | `Value::Array(非空)` → 各要素をコンテキストスタックにプッシュしてセクション本体を要素数分繰り返す |
| BR-2.3 | `Value::Object` → オブジェクトをコンテキストスタックにプッシュしてセクション本体を1回表示する |
| BR-2.4 | 上記以外の真値（`Bool(true)`, 非0の`Integer`/`Float`, 非空`String`）→ 元のコンテキストのままセクション本体を1回表示する |
| BR-2.5 | コンテキストスタック上にキーが存在しない場合 → BR-2.1と同様に非表示（「未定義」は単に偽として扱う。Q2=Aによりエラーにはならない） |

## 3. 逆セクションルール（FR-4）

| ルール | 内容 |
|---|---|
| BR-3.1 | セクションルール（BR-2.1〜BR-2.5）の真偽判定を反転する: BR-2.1に該当する場合のみ本体を表示し、それ以外は非表示 |

## 4. コンテキストスタック探索ルール

| ルール | 内容 |
|---|---|
| BR-4.1 | 変数・セクション名の解決は、コンテキストスタックの最も内側（最後にプッシュされた要素）から外側へ順に探索する |
| BR-4.2 | 内側のコンテキストにキーが存在すればそこで探索を終了する（外側に同名キーがあっても内側が優先される） |

## 5. パーシャルルール（FR-6）

| ルール | 内容 |
|---|---|
| BR-5.1 | `{{> name}}`に到達した時点で`PartialResolver::resolve(name)`を呼び出す（遅延評価、Application Design Q3=A） |
| BR-5.2 | 解決に失敗した場合、strictモードの設定に関わらず`RenderErrorKind::PartialNotFound`を返す（Q3=A） |
| BR-5.3 | 解決に成功した場合、得られた文字列を現在のデリミタ設定でParserにより再パースし、現在のコンテキストスタックでレンダリングする |
| BR-5.4 | パーシャルタグがスタンドアロン（行全体を占める）の場合、タグ直前の行頭空白を`indent`として記録し、パーシャル内容の各行（最終行の末尾改行がない場合を除く）の先頭に適用する |
| BR-5.5 | 現在解決中のパーシャル名チェーンを追跡し、新たに解決しようとする名前が既にチェーンに含まれる場合は`RenderErrorKind::PartialCycleDetected`を返す（Q4=B） |

## 6. コメント・デリミタ変更ルール（FR-4）

| ルール | 内容 |
|---|---|
| BR-6.1 | `{{! comment }}`は出力に一切寄与しない |
| BR-6.2 | `{{=<% %>=}}`形式のタグ以降、次のデリミタ変更まで新しいデリミタで以降のタグを解析する |
| BR-6.3 | デリミタ変更の有効範囲は現在パース中のテンプレート内に閉じる。パーシャル解決先の再パース（BR-5.3）は、パーシャル呼び出し時点のデリミタではなく、そのパーシャル自身の内容のデフォルトデリミタ（`{{` `}}`）から再開する |

## 7. スタンドアロンタグの空白トリミングルール

| ルール | 内容 |
|---|---|
| BR-7.1 | セクション開始・終了・逆セクション・パーシャル・コメント・デリミタ変更タグが、前後を空白のみで囲まれ、行全体を占める場合（スタンドアロン）、そのタグを含む行全体（前後の空白と改行）を出力から除去する |
| BR-7.2 | 変数展開タグ（`{{name}}`, `{{{name}}}`, `{{&name}}`）はスタンドアロン判定の対象外（常にインライン扱い） |

## 8. エラー処理ルール（Q5）

| ルール | 内容 |
|---|---|
| BR-8.1 | `ParseError`/`RenderError`は発生箇所の行番号・列番号（1始まり）を`line`/`column`フィールドに含める |
| BR-8.2 | パーシャル内で発生したエラーは、パーシャルファイル内での位置を報告する（呼び出し元テンプレートの行番号ではない） |
