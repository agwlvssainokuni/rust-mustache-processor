# Functional Design Plan — cli

`requirements.md`（FR-5〜FR-7）、`component-methods.md`、`unit-of-work-dependency.md`を踏まえた、cliユニットの詳細設計に向けた計画。

## 前提（要件で決定済み、質問不要）

- データのデフォルト入力元は標準入力、テンプレートは常にファイル引数（デフォルト経路）（FR-5）
- データファイルもコマンドライン引数で指定可能（FR-5。後述の通り、複数テンプレート対応により`--data <file>`フラグとして実現する）
- テンプレート・データ双方が同時に標準入力に指定された場合はエラー（FR-5）
- `--partials-dir`未指定時、テンプレートファイルが位置引数で指定されている場合はそのディレクトリをデフォルトとする（FR-6）
- 未定義変数はデフォルトで継続、strictモード相当のオプションでエラー化可能（FR-7、core-engineの`Mustache::with_strict`を呼び出すのみ）
- 出力先はファイル指定可能（FR-5）
- JSON/YAMLをサポート、TOML等は対象外（FR-3）
- 外部クレート: `clap`（引数解析）、`serde_json`/`serde_yaml`（データ変換）（`unit-of-work-dependency.md`）

## 決定が必要な論点（質問）

### Question 1: CLI引数の具体的な形状（複数テンプレート対応）

`component-methods.md`は`CliArgs::parse_args(argv) -> Result<CliArgs, CliArgsError>`という関数シグネチャのみを定義しており、具体的な引数の並び（位置引数かフラグのみか）は未決定。加えて、ユーザーからの追加要望により、テンプレートファイルを複数指定できるようにする（`cat`コマンドのように指定順で扱う）。

A) 位置引数 + フラグ方式、複数テンプレート対応: `mustache <template-file>... [-o/--output <file>] [--data <file>] [--partials-dir <dir>] [--strict] [--format json|yaml] [--template-stdin]`。テンプレートは1つ以上の位置引数（`Vec<PathBuf>`。`--template-stdin`指定時は位置引数を省略し標準入力を単一テンプレートとして扱う。両方指定時はエラー）。データは`--data <file>`指定時はそのファイル、未指定時は標準入力（デフォルト、FR-5）

B) フラグのみ方式: `mustache --template <file>... [--data <file>] [--output <file>] ...`。位置引数を一切使わず全て明示的なフラグで指定する。冗長だが曖昧さがない

C) Other（[Answer]: タグの下に詳細を記載）

[Answer]: A

**理由**: FR-5が例示するコマンド`mustache template.tmpl < data.json`は位置引数を前提とした形になっており、要件の書き方そのものが位置引数方式を想定していると読める。フラグのみ方式（B）は曖昧さがない一方、単純なユースケース（テンプレート1つ+標準入力データ）でも`--template`が必須になり冗長。Unix CLIツールの一般的な慣習にも合致する。

**実装時に発見した追加補正（要記録）**: テンプレートを複数位置引数（`Vec<PathBuf>`）にしたことで、FR-5が本来許容していた「データファイルを位置引数で指定」との両立が困難になった（何番目の位置引数がテンプレートで何番目がデータかclapで一意に判別できないため）。この矛盾をユーザーに確認し、データファイルは位置引数ではなく`--data <file>`フラグとして指定する方式を採用することで解消した（未指定時は従来通り標準入力がデフォルト）。「テンプレート・データ双方が同時に標準入力に指定された場合はエラー」（FR-5）は、`--template-stdin`指定かつ`--data`未指定（＝データも標準入力）の組み合わせに対して適用される。

**複数テンプレートの処理方式（ユーザー指定により確定）**: 生テキストを連結してから単一テンプレートとしてパースする方式（cat-then-process）ではなく、各テンプレートファイルを個別にパース・レンダリングしてから、レンダリング結果を指定順に連結する方式（process-then-cat）を採用する。

- **セパレータ**: 連結時にセパレータは挿入しない（`cat`と同様、レンダリング結果をそのまま結合）
- **`--partials-dir`未指定時のデフォルト解決（FR-6）**: 全ファイル共通ではなく、**各テンプレートファイルごとに、そのファイル自身のディレクトリ**をデフォルトとする（`--partials-dir`が明示指定されていれば、その値を全ファイル共通で使用する）
- **エラー時の挙動**: いずれか1ファイルでもパース・レンダリングエラーが発生した場合、全体を中断し出力は一切書き出さない（core-engineの「レンダリング全体を中断、部分的な出力は返さない」という既存方針との整合性を保つため。出力はメモリ上にバッファし、全テンプレートの処理が成功した時点で初めて書き出す）
- **副次的な利点**: 各テンプレートが独立してパースされるため、あるファイル内のデリミタ変更（`{{=<% %>=}}`）が別ファイルの解釈に漏れ出さない（生テキスト連結方式では発生し得た問題）

### Question 2: データ形式（JSON/YAML）判定の優先順位

`component-methods.md`の`DataLoader::detect_format(args) -> Result<DataFormat, DataLoaderError>`は判定に失敗しうることを示しているが、優先順位は未決定。

A) 明示`--format`オプションを最優先。未指定時はデータファイルの拡張子（`.json`/`.yaml`/`.yml`）で判定。データが標準入力等で拡張子が得られず`--format`も未指定の場合は`DataLoaderError`とする

B) 拡張子判定を優先し、`--format`は拡張子が無い・不明な場合のみのフォールバックとして使う

C) Other（[Answer]: タグの下に詳細を記載）

[Answer]: A

**理由**: 明示的なユーザー指定は常に推測より優先されるべきという原則は多くのCLIツール（`curl`, `docker`等）で共通。拡張子推測を優先すると、ユーザーが意図的に`--format`を指定しても無視されるような直感に反する挙動を招きうる。標準入力等で拡張子が得られずフォーマットも不明な場合にエラーとするのは、`component-methods.md`の`detect_format`が`Result`を返す設計とも整合する。

### Question 3: テンプレートの標準入力切替とパーシャルディレクトリのデフォルト解決

FR-5は「オプション指定時、テンプレート側を標準入力から読み込む切り替えが可能」としているが、具体的な指定方法とその際のパーシャルディレクトリのデフォルト解決（FR-6の「テンプレートファイルのディレクトリ」が存在しないケース）が未決定。

A) `--template-stdin`フラグで明示的に切り替える（この場合テンプレートの位置引数は省略。両方指定された場合はエラー）。`--partials-dir`未指定時は実行時のカレントディレクトリへフォールバックする

B) テンプレート位置引数に`-`を渡すUnix慣習を採用する。`--partials-dir`未指定時は同様にカレントディレクトリへフォールバックする

C) Other（[Answer]: タグの下に詳細を記載）

[Answer]: A

**理由**: FR-5の文言「オプション指定時」が文字通り明示的なオプションの存在を想定している。`-`慣習（B）はUnix文化として自然だが、位置引数の意味（テンプレートパスかstdin指示か）が値によって変わるため`clap`定義がやや複雑になり、エラーメッセージも分かりにくくなりやすい。カレントディレクトリへのフォールバックは、テンプレート由来のディレクトリが存在しないケースでの最も直感的な代替。なお`--template-stdin`はQuestion 1で確定した複数テンプレート指定とは併用できない（標準入力は単一ストリームのため、位置引数のテンプレートと同時指定された場合はエラーとする）。

### Question 4: 終了コード方針

`component-methods.md`の`CliRunner::run(argv) -> ExitCode`はコード体系までは規定していない。

A) 成功時0、それ以外は全て1とするシンプルな二値方式。エラー種別の詳細はstderrメッセージで判別する（Unix慣習、YAGNI）

B) エラー種別ごとに異なる非ゼロコード（引数エラー=2、入出力エラー=3、データ変換エラー=4、パース/レンダリングエラー=5等）を割り当て、シェルスクリプトからの分岐判定を可能にする

C) Other（[Answer]: タグの下に詳細を記載）

[Answer]: A

**理由**: 要件（FR-5〜FR-7）はエラー種別ごとの終了コード区別を一切要求していない。詳細なエラー原因はstderrメッセージで十分判別可能であり、v1のCLIツールとしてはYAGNI。将来的にスクリプト連携ニーズが明確になれば拡張すればよく、今から多段階のコード体系を固定すると後方互換性の制約が増える。

### Question 5: エラーメッセージの出力形式

A) `mustache: <message>`のようにプログラム名を前置した人間可読なプレーンテキストをstderrへ出力する（1〜数行）

B) JSON等の構造化形式でエラー詳細（種別・位置情報等）を出力する

C) Other（[Answer]: タグの下に詳細を記載）

[Answer]: A

**理由**: ローカルCLIツールとして人間が直接使うことを主眼とした設計（NFR-4がセキュリティ対象外、NFR-5が耐障害性対象外である点からも、エンタープライズ的な構造化ログ連携は想定していないと読める）。JSON構造化（B）は現時点で明確な消費者（ログ集約基盤等）が存在せず、過剰設計。
