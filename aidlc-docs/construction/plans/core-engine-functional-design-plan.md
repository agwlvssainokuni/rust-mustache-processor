## 前提（公式spec準拠、質問不要）

以下はNFR-2（公式mustache/specコンフォーマンス）により既に一意に決定されており、質問の対象としない:
- 変数展開のエスケープ規則（`&`, `<`, `>`, `"`, `'`をHTMLエンティティに変換、`{{{}}}`/`{{&}}`はエスケープなし）
- セクションの真偽判定（`false`/`null`/空配列 → 非表示、非空配列 → 各要素をコンテキストとして繰り返し、それ以外の真値 → 元のコンテキストのまま1回表示）
- 逆セクションの真偽判定（セクションの反転）
- コンテキストスタックのルックアップ順序（内側から外側へ探索）
- デリミタ変更のスコープ（現在のテンプレート内で変更後有効、パーシャル境界を越えない）
- スタンドアロンタグ（セクション・パーシャル・コメント・デリミタ変更が行全体を占める場合の空白・改行トリミング）
- パーシャルのインデント処理（パーシャルタグ前の空白を、挿入されるパーシャル内容の各行に適用）

## 決定が必要な論点（質問）

### Question 1: Value型における数値の内部表現
JSON/YAMLから読み込む数値データを、core-engineの`Value`型でどう表現するか。公式spec自体は数値の扱いを厳密には規定していない（テストケースは主に文字列・真偽値・配列・ハッシュが中心）。

A) 整数・浮動小数点数を区別せず、単一の`f64`（または文字列保持）で統一的に扱う。シンプルだが`1`が`"1"`ではなく`"1.0"`のように出力される可能性がある

B) 整数（`i64`）と浮動小数点数（`f64`）を別バリアントとして区別し、それぞれ自然な文字列表現（`1` → `"1"`, `1.5` → `"1.5"`）でレンダリングする

C) Other（[Answer]: タグの下に詳細を記載）

[Answer]: B

**理由**: `{{ count }}`で`count=5`のとき`"5"`ではなく`"5.0"`と出力されるのは直感に反する。JSON（`serde_json::Number`は`is_i64()`/`is_f64()`で区別可能）・YAMLとも整数/浮動小数点数を区別しており、その情報を`Value`型でも保持する方が自然な文字列表現を素直に実現できる。

### Question 2: strictモードの適用範囲
FR-7は「未定義変数の参照」についてstrictモードでのエラー化を定めているが、パーシャル未解決・セクション内の未定義参照など、他の「未定義」ケースにも同様に適用すべきか。

A) 変数展開（`{{var}}`, `{{{var}}}`）の未定義参照のみが対象。セクション（真偽判定に使われるだけなので「未定義」は単に偽として扱われ、そもそもエラー概念がない）やパーシャル未解決は別ルール（Question 3参照）とする

B) 変数展開に加え、セクション評価時にコンテキストが完全に見つからない場合も含め、あらゆる「未定義参照」をstrictモード時にエラーとする

C) Other（[Answer]: タグの下に詳細を記載）

[Answer]: A

**理由**: セクションにおける「キーが存在しない」は仕様上そもそも「偽」として扱われる正常な状態であり、「未定義」というエラー概念が存在しない。ここにstrictモードのエラー化を持ち込むと、テンプレート再利用時に意図的に省略されたキー（条件によって省略されるフィールド等）が軒並みエラーになり、Mustacheのセクションの本質的な使い方と矛盾する。FR-7の文言も「未定義変数の参照」に限定している。

### Question 3: パーシャル未解決時のデフォルト挙動
Application Design Q3（遅延評価）により「パーシャルタグに到達した時点でエラー判定」までは決定済み。到達時、strictモードの有無によってどう挙動を変えるか。

A) strictモードに関わらず、パーシャルが見つからない場合は常にエラーとする（ファイルの存在は実行環境の問題であり、変数の欠如とは性質が異なるため）

B) 未定義変数と同様に、strictモードがオフなら空文字として継続、オンならエラーとする（FR-7の挙動をパーシャルにも統一適用）

C) Other（[Answer]: タグの下に詳細を記載）

[Answer]: A

**理由**: パーシャルファイルが見つからないのは「データに特定のキーがない」という日常的なケースとは性質が異なり、多くの場合パーシャル名のタイプミスやディレクトリ指定ミスといった設定不備を示す。これを未定義変数と同様に暗黙的に空文字へフォールバックさせると、バグが出力に表れず気づきにくくなる。ファイルI/Oの失敗は早期に顕在化させる方が安全。

### Question 4: パーシャルの再帰・循環参照の扱い
パーシャルが直接的・間接的に自分自身を参照する場合（例: `a.mustache`が`{{> b}}`を含み、`b.mustache`が`{{> a}}`を含む）の挙動。

A) 特別な検出は行わず、Rustのコールスタックが尽きるまで再帰的に解決を試みる（実行時にスタックオーバーフローで異常終了する可能性がある）

B) パーシャル解決チェーンを追跡し、循環を検出した時点で`RenderError`として明示的にエラーを返す

C) Other（[Answer]: タグの下に詳細を記載）

[Answer]: B

**理由**: 検出なしで再帰させると、Rustではスタックオーバーフロー時にプロセスが異常終了し（`panic`として捕捉できない）、呼び出し側が`Result`でハンドリングできない。core-engineは既に`ParseError`/`RenderError`による`Result`ベースのエラー設計を採用しており、循環検出はその設計と一貫性がある。実装コストも解決チェーンを`Vec`等で追跡するだけで小さい。

### Question 5: エラー情報の詳細度
`ParseError`/`RenderError`に含める情報の粒度。

A) エラーの種別・メッセージのみ（例: "unexpected end of section"）。実装をシンプルに保つ

B) エラー種別・メッセージに加え、テンプレート内の行番号・列番号（または文字位置）を含める。デバッグ時の利便性が高いが、Parser/Rendererの実装が複雑化する

C) Other（[Answer]: タグの下に詳細を記載）

[Answer]: B

**理由**: Parserはテンプレートを文字単位で走査するため、行・列のトラッキングは実装コストが小さい一方、テンプレート作成者にとってのデバッグ効率は大きく向上する。ライブラリとしての利用者体験（エラーメッセージの有用性）に直結する部分であり、投資対効果が高い。
